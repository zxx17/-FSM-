赛题名称：
基于FSM状态机的流程管理系统

赛题简介：
随着电子商务的蓬勃发展，快递物流行业面临着前所未有的挑战和机遇。顺丰快递，作为行业的领军企业，一直致力于通过技术创新提升服务质量和运营效率。本次赛题聚焦于快递物流中的核心环节——快递状态管理，要求参赛者利用有限状态机（FSM）的概念，设计并实现一个高效、可靠的快递状态管理系统。


赛题背景：
在快递物流过程中，一个包裹会经历多个状态，如待揽收、已揽收、运输中、派送中、已签收、已拒收、异常件、已完成等。有效的状态管理对于确保包裹准时、安全地送达至关重要。然而，现有的状态管理系统往往存在效率低下、错误率高等问题。为了解决这些问题，顺丰快递希望通过引入FSM状态机来优化状态流转的逻辑，提高整个物流过程的自动化和智能化水平。

赛题要求：
工程化思维：本赛道需要选手有一定的工程化思维能力，选手应该以做工程的角度去完成比赛，将校园所学和实际工程结合，为自己将来的工作做好准备。让项目的完成不仅仅是“能够运行”，而是“良好的工程”。
自定义业务APP：创建一个工作流的业务APP，APP上包含工作流的状态、角色、角色可以操作的状态。
自定义业务流程：根据业务APP,创建一个业务流程，业务流程由一系列步骤组成，每个步骤可以有一个或多个相互关联的状态。
状态转移：定义在步骤中状态之间的转换方式，每个状态转换可能会有一些触发条件/规则，并可能导致一些行为。
角色管理：每个流程涉及到的人员可能会有不同的角色，角色之间可以有一定的层级关系。
流程实例：可以创建流程的实例，并根据流程定义来执行状态转换。

功能点：
1、模拟多种角色参与的,与多种状态交互的复杂流程。
2、提供流程管理的增、删、查三种自动化测试用例；
3、提供模拟流程执行的过程测试用例；

示例场景：
假设我们正在开发基于FSM状态机的物流运转平台，转运流程包含以下状态：
待揽收(PENDING)
已揽收(COLLECTED)
待支付 (PAYING)
已支付 (PAID)
运输中 (TRANSITING)
派送中 (DELIVERY)
已签收 (DELIVERID)
已拒收 (REFUSED)
异常件 (EXCEPTION)
已取消 (CANCELLED)
已完成 (COMPLETED)

事件包括：
揽收（COLLECT）
支付（PAY）
发货（SHIP）
取消（CANCEL）
自动执行（AUTO）
派送（DELOVERY）

人员角色包含：
小哥（POSTMAN）
寄件人 (SENDER)
转运员 (TRANSITER)
司机 (DRIVER)
收件人 (RECIPTENT)

人员角色权限：
小哥：待揽收 ->已揽收、运输中 -> 派送中 、派送中 -> 异常件
寄件人：待支付 -> 已支付、待揽收 ->已取消、待支付 -> 已取消
转运员 ：已支付 ->运输中
司机 ：已支付 ->运输中
收件人 ：派送中 -> 已签收、派送中 ->已拒收
AUTO：已签收 -> 已完成 、已取消  -> 已完成、已拒收 -> 已完成、异常件 -> 已完成


可能的流程包括：
1、待揽收 ->已揽收 -> 待支付 -> 已支付 ->运输中 -> 派送中 -> 已签收 -> 已完成
2、待揽收 ->已揽收 -> 待支付 -> 已取消  -> 已完成
3、待揽收 ->已揽收 -> 待支付 -> 已支付 ->运输中 -> 派送中 ->已拒收 -> 已完成
4、待揽收 ->已揽收 -> 待支付 -> 已支付 ->运输中 -> 派送中 -> 异常件 -> 已完成
